<objective>
## Goal
Migrate saved suspension setups from localStorage to Supabase — each logged-in user gets up to 3 cloud-synced setups that persist across devices.

## Purpose
This is the core feature that makes accounts worthwhile. Riders can set up their suspension at home and pull it up on the trail from any device.

## Output
- `saves` table in Supabase with Row Level Security (users can only see their own data)
- Save, load, edit, delete all talking to Supabase instead of localStorage
- UI unchanged — same save strip, same saved cards, same edit flow
</objective>

<acceptance_criteria>

## AC-1: Saves persist in Supabase per user
```gherkin
Given a logged-in user saves a suspension setup
When they refresh the page or open the app on another device
Then their saved setups are loaded from Supabase, not localStorage
```

## AC-2: Max 3 saves enforced
```gherkin
Given a user already has 3 saved setups
When they try to save a 4th
Then the save is blocked and an appropriate message is shown
```

## AC-3: Users can only see their own saves
```gherkin
Given two different users are logged in on different devices
When each views their saved setups
Then each sees only their own data (RLS enforced at DB level)
```

## AC-4: Edit and delete work against Supabase
```gherkin
Given a user edits or deletes a saved setup
When the action completes
Then the change is reflected in Supabase and persists on refresh
```

<task type="auto">
  <name>Task 1: Generate Supabase SQL for saves table + RLS</name>
  <files>.paul/phases/03-cloud-saves/saves-schema.sql</files>
  <action>
    Create a SQL file with the complete schema and RLS policies. This will be run by the user in the Supabase SQL editor.

    The SQL should:
    1. Create the saves table:
       ```sql
       create table saves (
         id uuid default gen_random_uuid() primary key,
         user_id uuid references auth.users(id) on delete cascade not null,
         name text not null,
         fork_brand text,
         fork_model text,
         shock_brand text,
         shock_model text,
         weight_kg numeric,
         ride_style text,
         damping_snapshot jsonb,
         created_at timestamptz default now(),
         updated_at timestamptz default now()
       );
       ```
    2. Enable RLS on the table
    3. Add policy: users can SELECT their own rows (user_id = auth.uid())
    4. Add policy: users can INSERT rows where user_id = auth.uid()
    5. Add policy: users can UPDATE their own rows
    6. Add policy: users can DELETE their own rows
  </action>
  <verify>File .paul/phases/03-cloud-saves/saves-schema.sql exists and contains valid SQL.</verify>
  <done>SQL ready for user to run in Supabase</done>
</task>

<task type="auto">
  <name>Task 2: Replace localStorage save logic with Supabase in index.html</name>
  <files>index.html</files>
  <action>
    Replace the existing localStorage-based save/load/edit/delete functions with Supabase equivalents.

    Key changes:
    1. **renderSaves()** — instead of reading from localStorage, query Supabase:
       ```js
       const { data } = await supabaseClient.from('saves').select('*').order('created_at');
       ```
       Render the returned rows into save cards. Handle empty state same as before.

    2. **confirmSave()** — instead of writing to localStorage:
       - If creating new: INSERT row into saves (check count ≤ 3 first)
       - If editing (editingId set): UPDATE the existing row by id
       ```js
       await supabaseClient.from('saves').upsert({ id: editingId, user_id: currentUser.id, name, fork_brand, ... })
       ```

    3. **deleteSave(id)** — instead of splicing localStorage array:
       ```js
       await supabaseClient.from('saves').delete().eq('id', id)
       ```

    4. **startEditName(id)** — load setup from the Supabase row (already rendered in cards), populate form fields same as before.

    5. Track current user: in onAuthStateChange, store `session.user` in a `currentUser` variable at the top of the script.

    6. Call `await renderSaves()` after login (inside onAuthStateChange when session exists) and after every save/delete/edit.

    7. Make renderSaves async — all DB calls need await.

    Important:
    - Keep the save card HTML structure identical — same classes, same edit/delete buttons
    - Keep the 3-save limit UI the same (hide save strip when 3 saves exist)
    - Do NOT remove localStorage entirely yet — saved weight (loadSavedWeight) still uses it
    - Avoid: running renderSaves before auth state is confirmed (causes empty flash)
  </action>
  <verify>

  </verify>
